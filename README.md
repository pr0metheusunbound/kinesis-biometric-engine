# Kinesis: Biometric Anti-Fraud Engine

### Комплексная система поведенческой детекции ботов на основе биомеханического анализа и криптографической верификации сессий

Traditional anti-fraud systems rely on static properties: IP addresses, HTTP headers, and browser fingerprints (Canvas, WebGL, User-Agent). However, advanced automated scripts (Puppeteer, Playwright, Selenium, AI-driven scrapers) can perfectly spoof these fingerprints.

**Kinesis** takes a fundamentally different approach. It analyzes the **physics of interaction** — the cognitive-motor patterns of a real human being. A human being has muscle mass, physical inertia, and neuromuscular tremor, which cannot be perfectly simulated by traditional mathematical functions without triggering anomalies.

## 1. Сквозная архитектура системы (End-to-End Flow)

Система Kinesis состоит из трех изолированных и архитектурно независимых слоев:

1. **Frontend Instrumentation (`kinesis.js`):** Легковесный модуль на стороне клиента (браузера), который собирает сырые координаты мыши, клики и временные метки, минимизируя нагрузку на CPU пользователя.
    
2. **Ingestion Gateway (FastAPI):** Высокопроизводительный асинхронный бэкенд, который отвечает за валидацию данных, криптографическую защиту от подмены сессий и буферизацию логов.
    
3. **Analytics Core (NumPy / Pandas):** Математический движок, который разбирает накопленные треки, строит физические векторы и выносит вердикт (`HUMAN` или `BOT`).
    

## 2. Глубокий технологический разбор компонентов

### Слой 1: Клиентская телеметрия (`kinesis.js`)

Главная инженерная сложность сбора координат — не перегрузить Event Loop браузера. Стандартное событие `mousemove` может генерировать сотни вызовов в секунду, вызывая лаги интерфейса (Jank).

- **Механизм Throttling (Дросселирование):** Скрипт перехватывает координаты с фиксированным шагом (например, `50 мс`), используя встроенные таймеры высокой точности (`performance.now()`). Этого дискретного шага более чем достаточно для математического восстановления кривой.
    
- **Аппаратная инвариантность (Нормализация):** Сырые координаты пикселей (`clientX`, `clientY`) зависят от разрешения монитора пользователя. Модель, обученная на разрешении 1920x1080, сломается на экране смартфона или 4K-мониторе. `kinesis.js` на лету нормализует координаты в диапазон $[0.0, 1.0]$ относительно текущей ширины и высоты вьюпорта браузера:
    
    $$\text{normalized\_x} = \frac{\text{clientX}}{\text{window.innerWidth}}$$
    
- **Гарантия доставки при уходе (Emergency Flush):** Если бот совершил целевое действие (например, заполнил форму) и мгновенно закрыл вкладку, стандартный асинхронный `fetch` будет оборван браузером. Kinesis решает эту проблему отправкой финального батча через `fetch()` с флагом `keepalive: true` (эволюция `navigator.sendBeacon`), что заставляет сетевой стек ОС гарантированно завершить отправку даже после уничтожения процесса страницы.
    

### Слой 2: Приёмный шлюз (FastAPI API Security)

При проектировании антифрода нельзя слепо доверять данным от клиента. Злоумышленник может перехватить структуру JSON-пакета и отправить искусственно сгенерированные "человеческие" координаты напрямую в API бэкенда, минуя браузер.

- **Защита от Spoofing (Cryptographic Handshake):** Kinesis запрещает бесконтрольный прием телеметрии. Перед началом сбора данных фронтенд обязан выполнить «рукопожатие» (`/api/v1/handshake`), передав параметры экрана. Сервер генерирует уникальный `sessionId` и подписывает его криптографическим ключом на базе алгоритма **HMAC-SHA256**.
    
- **Сессионный токен:** Клиент получает токен, содержащий `sessionId` и метку времени (timestamp). При каждой отправке батча телеметрии в заголовке `X-Kinesis-Session` передается этот токен. Сервер на лету пересчитывает HMAC-хэш. Если токен изменен, подделан или его время жизни истекло, данные отбрасываются (HTTP 401/403).
    
- **Асинхронный I/O буфер:** Чтобы бэкенд не блокировался при записи логов на диск, вызовы стандартного потокобезопасного логгера делегируются в пул потоков через `asyncio.to_thread`. Это предотвращает деградацию производительности (RPS) при массовых атаках ботов.
    

### Слой 3: Аналитическое ядро и математические фичи (The Math Core)

Магическая часть Kinesis — это перевод сырых массивов координат $(x, y, t)$ в физические признаки, описывающие поведение живой материи.

После восстановления дельт движений ($\Delta x, \Delta y, \Delta t$), движок рассчитывает три фундаментальные метрики:

#### 1. Энтропия Шеннона по углам направлений (Directional Entropy)

Для каждого шага вычисляется угол направления движения курсора в радианах через арктангенс:

$$\theta_i = \text{atan2}(\Delta y_i, \Delta x_i)$$

Все углы распределяются по 16 частотным корзинам (bins) от $-\pi$ до $\pi$. Затем вычисляется информационная энтропия Шеннона:

$$H(X) = -\sum_{i=1}^{n} P(x_i) \log_2 P(x_i)$$

- **Логика детекции:** Обычные боты двигаются по идеальным прямым линиям. Все их углы попадают в 1–2 корзины. Их энтропия критически низкая ($< 2.0$). Человек постоянно совершает микро-корректировки траектории, его углы распределены хаотично, что дает высокую энтропию ($> 3.0$).
    

#### 2. Межквартильный размах скорости (Velocity IQR)

Вместо стандартной дисперсии скорости, которая уязвима к аномальным скачкам (например, телепортации курсора из-за лага отрисовки), Kinesis использует робастную метрику — **IQR (Interquartile Range)**:

$$\text{IQR} = Q_3 - Q_1$$

Где $Q_3$ — 75-й перцентиль скорости, а $Q_1$ — 25-й перцентиль.

- **Логика детекции:** Боты автоматизации часто двигаются с фиксированной программной скоростью, заложенной разработчиком (например, шаг 20 пикселей каждые 20 мс). Их `Velocity IQR` стремится к нулю. Человек подчиняется биомеханическому закону Фиттса: сначала идет мощный баллистический рывок к цели (высокая скорость), а затем фаза точного прицеливания (резкое торможение). Это создает огромный здоровый разброс скоростей (`IQR` у человека в сотни раз выше, чем у стандартного бота).
    

#### 3. Автокорреляция первого порядка (Lag-1 Autocorrelation of Delta-X)

Это главный рубеж защиты от продвинутых ботов. Когда разработчики ботов понимают, что их ловят на идеальных прямых линиях (низкой энтропии), они начинают подмешивать к координатам случайный математический шум через генераторы случайных чисел (`random.uniform()`, `random.randint()`). Энтропия такого бота взлетает, и он обходит базовые проверки.

Kinesis закрывает эту слепую зону через оценку коэффициента автокорреляции временного ряда дельт координат с лагом в 1 шаг:

$$\rho_1 = \frac{\sum_{i=2}^{n} (\Delta x_i - \bar{\Delta x})(\Delta x_{i-1} - \bar{\Delta x})}{\sum_{i=1}^{n} (\Delta x_i - \bar{\Delta x})^2}$$

- **Логика детекции:** Случайный шум программного генератора — это **Белый шум**. Каждый следующий шаг генерации математически независим от предыдущего. Из-за дискретности пиксельной сетки автокорреляция белого шума падает в отрицательную зону (в районе $-0.1 \dots -0.5$).
    
    Рука человека обладает **физической массой и инерцией**. Мы не можем физически телепортировать руку влево, если в предыдущую миллисекунду двигали ее вправо. Человеческие дельты сильно скоррелированы между собой. У живого пользователя `Dx_Autocorr` всегда находится в стабильной **положительной** зоне ($0.25 \dots 0.7$). Отрицательная или нулевая автокорреляция — это мгновенный и неоспоримый маркер искусственного шума программного скрипта.
    

## 3. Моделирование угроз и валидация (Adversarial Testing)

Проект содержит встроенный модуль тестирования угроз на базе фреймворка **Playwright**. Скрипт-симулятор выполняет роль "атакующего агента" и имеет два режима:

1. **Линейный бот (Наивный скрипт):** Генерирует математически безупречные прямые линии между элементами интерфейса. Мгновенно блокируется по метрике `Entropy`.
    
2. **Шумящий бот (Продвинутый скрипт):** Имитирует человеческий тремор, накладывая псевдослучайный высокочастотный джиттер на каждый шаг движения. Успешно обходит энтропийный фильтр, но **со стопроцентной точностью перехватывается аналитическим ядром** на этапе проверки `Dx_Autocorr`, так как физика его движений выдает природу белого шума.
    

## 4. Архитектурные компромиссы и точки роста (Future Roadmap)

Kinesis спроектирован с учетом требований к производительности, однако, как и любая реальная система, имеет инженерные компромиссы:

- **Текущее ограничение:** Система работает в асинхронном режиме (оффлайн-анализ накопленных логов с помощью Pandas). Это эффективно для постаудита и расследования инцидентов (SOC).
    
- **Точка развития (Реалтайм):** Для блокировки ботов непосредственно в момент клика по кнопке (Inline-детекция), аналитическое ядро должно быть интегрировано прямо внутрь FastAPI эндпоинта. Для этого алгоритмы вычисления энтропии и автокорреляции уже оптимизированы и переведены с тяжеловесного Pandas на чистый, оптимизированный по памяти **NumPy**, выполняющий операции на уровне C-массивов за микросекунды.
    
- **Эволюция векторов атак:** Против продвинутых ботов, использующих уравнения случайного блуждания или шум Перлина, планируется внедрение третьей производной расстояния — **анализа рывков (Jerk Analysis)** и быстрого преобразования Фурье (FFT) для валидации частотного спектра человеческого тремора (8–12 Гц).
    

### Как развернуть и протестировать систему:

1. **Запуск API:** `uvicorn app:app --reload --port 8000`
    
2. **Сбор данных:** Открой тестовый `index.html` в браузере, поводи мышкой, сделай клики. Данные автоматически соберутся в `kinesis_telemetry.log`.
    
3. **Запуск атаки:** Запусти `python bot_simulator.py` для генерации бот-трафика.
    
4. **Анализ:** Запусти `python analyze_telemetry.py` для вывода финального отчета с вердиктами.
